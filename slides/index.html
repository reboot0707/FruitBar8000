8000

5 分鐘專案成果報告

約 30 秒。FruitBar8000 的名字來自 Foobar2000；這次不是要做完整播放器，而是先把音樂 metadata 的資料關係與 CRUD 基礎打穩。

01 · SCOPE

先解決「資料怎麼放、功能怎麼長」

今天的範圍很明確：

  • 建模：歌曲、專輯、創作者、曲風
  • 維護：新增、查詢、修改、軟刪除
  • 呈現：以 WinForms 完成操作介面

先讓核心資料可信，才有條件延伸播放、封面與 Web API。

約 40 秒。這一階段聚焦資料表設計、關鍵 metadata 的 CRUD，以及能操作這些功能的 WinForms。播放器、串流整合都先放到後續，避免範圍失控。

02 · DATA MODEL

多對多關聯，才裝得下真實的音樂資料

歌曲↔創作者 ArtistsSongs
歌曲↔專輯 SongsAlbums

一首歌可有多位創作者，也可透過中介表保留角色與曲序；資料模型不必被單一情境綁死。

約 50 秒。音樂資料不是一首歌只對一位歌手。透過 ArtistsSongs、SongsAlbums 等中介表，可以表達多位創作者、不同 credit role、同一專輯的曲序，也讓後續功能有擴充空間。

UI 不直接承擔資料規則，服務也不綁死某一個畫面。

約 50 秒。這裡不是嚴格宣稱已完成教科書式三層架構，而是採「類三層式」維護：畫面處理互動，Service 集中 CRUD，EDMX 管理資料關係。核心價值是降低耦合。

把「方法」當參數：新增查詢類型時，保留共用的輸入判斷與 DataGridView 綁定。

約 50 秒。三種查詢最後都回傳相同的 SongQueryResult。畫面用 Func 把選到的查詢方法傳給 CallQuery，因此關鍵字整理、結果綁定與欄名設定只寫一次。

05 · MAINTAINABILITY

資料安全與可遷移性，也放進第一版設計

↶

軟刪除保留資料

刪除歌曲時設定 IsDeleted = true；一般查詢主動排除，降低誤刪風險。

⇄

結構可重建、可移動

資料表規格與關聯同步保存在 dbscript/，不只存在單一資料庫環境。

約 40 秒。第一個是軟刪除，程式不直接移除資料列，而是標記 IsDeleted。第二個是把 SQL 腳本納入專案，讓資料庫結構可以追蹤與遷移。兩者都是為了維護性。

06 · NEXT

從可靠的 metadata 核心，走向真正可用的音樂服務

  1. 01媒體連結查到歌曲後，開啟合法串流平台資訊
  2. 02專輯封面利用預留欄位，從介面維護封面
  3. 03資料匯入串接合法 metadata 來源，提升實用性
  4. 04Web API抽取元件、邏輯與資料架構，服務更多前端

FruitBar8000 的第一步：不是播放更多歌，而是先把每首歌的資料關係做好。

約 40 秒。後續順序是媒體連結、專輯封面、合法 metadata 匯入，最後把既有的資料與業務邏輯抽成 Web API。收尾強調：這一版的成果是可擴充的資料核心。
